## 4. Solution Strategy

Tóm tắt các quyết định kiến trúc quan trọng (chi tiết tại Section 9):

- **SSG + Sync Pipeline** — Content được sync từ Notion thành MDX files trong Git trước khi build; `next build` chỉ đọc local MDX, không gọi Notion API. Giải quyết triệt để timeout và đảm bảo build deterministic (AD-03).
- **Dual-trigger sync** — Notion Webhook làm primary trigger (real-time, vài phút); Vercel Cron hằng ngày làm lưới an toàn. Đảm bảo cả speed (REQ-FUNC-014) và consistency (REQ-FUNC-015) (AD-04).
- **Client-side static search** — Pagefind xây search index lúc build; query 100% phía client. Không cần backend search, không có server latency (AD-05).
- **Vercel all-in-one** — KV (view counter), Blob (ảnh từ Notion), Analytics, Cron, Image Optimization — toàn bộ trên Vercel Free tier, không cần service riêng (AD-02, AD-08, AD-09).
- **PWA với Serwist** — Service Worker quản lý offline cache với stale-while-revalidate và expiration (tối đa 30 bài); update tự động sau deploy (AD-07).
- **Atomic sync per bài** — Mỗi bài sync là đơn vị atomic; lỗi ở bất kỳ bước nào → abort bài đó, không ghi file bán phần; blog vẫn phục vụ nội dung cũ (AD-03).

### 5.2 Whitebox Level 2 — Sync Pipeline

Sync Pipeline là block phức tạp và rủi ro cao nhất — drill down:

| Sub-block | Trách nhiệm | Ghi chú |
|-----------|-------------|---------|
| `client.ts` | Khởi tạo Notion client với API key từ env | Singleton |
| `convert.ts` | Fetch blocks từ Notion → markdown string (dùng `notion-to-md`) | Dùng lại cho cả sync và draft preview |
| `images.ts` | Download ảnh từ signed URL Notion → upload Vercel Blob → trả URL stable | Xử lý cover image và inline images |
| `syncPost.ts` | Orchestrate 1 bài: fetch → convert → images → build frontmatter → diff → commit | Atomic: lỗi bất kỳ bước → abort, không ghi file |
| `syncAll.ts` | Query Notion chỉ bài có `last_edited_time > sync:last_run` (KV) → chạy syncPost → gộp 1 commit → cập nhật `sync:last_run` | 1 bài lỗi không block bài khác; dùng mutex `sync:running` trong KV để chặn concurrent runs |

## 6. Runtime View

### Kịch bản 1: Người đọc đọc bài viết (happy path)

```
Người đọc → Browser → Vercel CDN → Next.js (SSG)
```

1. Người đọc truy cập `/posts/nextjs-performance`
2. Vercel CDN phục vụ HTML tĩnh đã pre-render (không cần server compute)
3. Browser nhận HTML → render nội dung ngay (LCP target ≤ 2.5s)
4. Browser load JS bundle → hydrate client components (ThemeToggle, SearchBox)
5. Khi người đọc cuộn xuống khu vực bình luận → IntersectionObserver trigger → Giscus embed load
6. Client component gửi POST `/api/view` → Vercel KV tăng counter lên 1

*Notion API không được gọi ở bất kỳ bước nào.*

---

### Kịch bản 2: Tác giả publish bài mới (sync flow)

```
Notion → Webhook → Sync Pipeline → GitHub → Vercel Deploy → Web
```

1. Tác giả đổi Status bài từ Draft → Published trên Notion
2. Notion gửi HTTP POST đến `/api/notion-webhook` với payload và signature header
3. Handler verify signature → hợp lệ → extract `pageId`
4. `syncPost(pageId)` bắt đầu:
   - Fetch page properties từ Notion API
   - Fetch blocks → convert markdown (`notion-to-md`)
   - Phát hiện ảnh Notion signed URL → download → upload Vercel Blob → replace URL trong markdown
   - Build MDX frontmatter (title, slug, tags, date, excerpt, cover, notionLastEditedTime)
   - Diff với file MDX cũ → có thay đổi → ghi file
   - Commit `content/posts/{slug}.mdx` qua GitHub Contents API
5. GitHub commit trigger Vercel deploy → `next build` chạy từ MDX local
6. Pagefind rebuild search index → Serwist rebuild service worker
7. Deploy xong → bài xuất hiện tại `/posts/{slug}`

*Tổng thời gian target: ≤ 5 phút từ bước 1 đến bước 7.*

---

### Kịch bản 3: Sync lỗi — Notion API trả về 503

```
Cron → syncAll → Notion API 503 → abort bài đó → blog phục vụ nội dung cũ
```

1. Vercel Cron trigger POST `/api/cron-sync` (2:00 AM hằng ngày)
2. Handler verify `Authorization: Bearer <CRON_SECRET>` → gọi `syncAll()`
3. `syncAll()` fetch danh sách bài `Published` và `Archived` từ Notion
4. Bài thứ 2 trong batch: Notion API trả về 503
5. `syncPost` bắt exception → log lỗi (pageId, slug, error message) → skip bài này
6. Các bài còn lại tiếp tục được xử lý bình thường
7. Nếu có bài thay đổi → commit 1 lần duy nhất cuối batch
8. File MDX của bài thứ 2 **không bị thay đổi** → blog vẫn phục vụ nội dung cũ của bài đó

---

### Kịch bản 4: Tác giả xem bài draft

1. Tác giả truy cập `/api/draft?secret=<DRAFT_SECRET>&slug=<slug>`
2. Handler so sánh `secret` với `process.env.DRAFT_SECRET`
3. Secret đúng → `draftMode().enable()` → redirect sang `/posts/{slug}`
4. Page handler detect `draftMode().isEnabled === true`
5. Fetch bài trực tiếp từ Notion API (bao gồm Draft) → render live (không dùng MDX cache)
6. Hiển thị banner "Preview Mode" để tác giả biết đây là preview

---

### Kịch bản 5: Tác giả xem bài Archived qua debug URL

1. Tác giả truy cập `/posts/bai-da-archive?debug=<DRAFT_SECRET>`
2. Page handler `/posts/[slug]` đọc `searchParams.debug`
3. So sánh `debug` với `process.env.DRAFT_SECRET` → khớp
4. Đọc MDX file (status: archived) → render nội dung đầy đủ
5. Hiển thị ribbon "Archived" ở đầu trang
6. Nếu `debug` sai hoặc không có → redirect 302 về `/` (xử lý như người đọc thông thường)

---

### Kịch bản 6: Người đọc truy cập URL bài Archived

1. Người đọc truy cập `/posts/bai-da-archive` (không có `?debug=`)
2. Page handler đọc MDX → phát hiện `status: archived`
3. Không có `debug` param hợp lệ → `redirect('/', 302)`
4. Người đọc về trang chủ, không thấy nội dung bài

---

### Kịch bản 7: Người đọc mở bài đã xem khi offline

1. Thiết bị mất mạng; người đọc mở `/posts/nextjs-performance` (đã xem trước)
2. Browser gửi request → Service Worker intercept
3. SW kiểm tra runtime cache (`posts-cache`) → tìm thấy response cho URL này
4. SW trả response từ cache → trang hiển thị bình thường
5. (Khi có mạng lại) SW revalidate cache ở background

## 8. Crosscutting Concepts

### 8.1 Security

**Webhook authentication:** Mọi request đến `/api/notion-webhook` phải verify HMAC signature trong header trước khi xử lý. Request không có signature hoặc signature sai → trả về 401, không thực thi logic, ghi log warning.

**Draft preview:** Secret so sánh bằng constant-time comparison để tránh timing attack. Secret ≥ 32 ký tự, lưu trong env var, không bao giờ xuất hiện trong response hay commit history.

**Cron authentication:** `/api/cron-sync` verify `Authorization: Bearer <CRON_SECRET>` — ngăn trigger từ bên ngoài Vercel.

**Secret management:** Tất cả secrets qua Vercel Environment Variables. `.env.example` chỉ có placeholder. Pre-commit hook (optional) ngăn commit file `.env` thực.

**Newsletter:** Email validate tại client (UX) và tại server (security) trước khi gọi Resend API. Endpoint có rate limiting (nếu Vercel plan hỗ trợ) hoặc honeypot field để giảm spam.

### 8.2 Error Handling

**Sync errors (per-bài):** Mỗi bước trong `syncPost` wrapped trong try/catch riêng. Lỗi ở bất kỳ bước → throw lên caller → abort toàn bài → không ghi file MDX bán phần → ghi log → trả về error status.

**Sync errors (batch):** Trong `syncAll()`, mỗi `syncPost()` call wrapped trong try/catch riêng. Lỗi 1 bài → log + continue → không abort batch.

**Vercel Function timeout:** Sync 1 bài target ≤ 10s để không bị Vercel cut off (Hobby tier: 10s max). Nếu vượt → bài không được commit → cron sẽ retry lần sau.

**Build failure:** Nếu Git commit từ sync trigger build và build fail → Vercel tự động rollback về deployment trước. Site không bị hỏng.

### 8.3 Logging & Observability

**Sync logs:** Mọi lần sync (thành công/thất bại) ghi JSON log với fields:
```json
{
  "timestamp": "ISO-8601",
  "trigger": "webhook | cron",
  "pageId": "notion-page-id",
  "slug": "post-slug",
  "result": "success | failed | skipped",
  "durationMs": 1234,
  "error": "error message nếu có"
}
```
Secrets và token KHÔNG được ghi vào log.

**Web analytics:** Vercel Analytics (embed tự động) → page views, unique visitors, sessions, referrer. Privacy-friendly — không cần GDPR cookie consent.

**View counter:** Mỗi POST `/api/view` tăng `views:<slug>` trong KV. `syncAll` đọc KV → ghi ngược `viewCount` vào MDX frontmatter để sort-by-popular chạy static.

### 8.4 Caching Strategy

| Layer | Mechanism | TTL / Policy |
|-------|-----------|-------------|
| Static pages (SSG) | Vercel CDN | Immutable per deploy; invalidate on new deploy |
| Images (Blob) | Vercel Blob CDN | Permanent (URL includes content hash) |
| `next/image` | Vercel Image Optimization | Default CDN caching |
| Runtime posts (SW) | Service Worker Cache API | Stale-while-revalidate; max 30 entries (LRU eviction) |
| App shell (SW) | Precache (Serwist) | Updated on new SW deploy |

### 8.5 Performance Patterns

- **Server Components mặc định** — tất cả components là Server Component trừ những cái cần browser API. Giảm JS bundle gửi xuống client.
- **`use client` chỉ cho:** ThemeToggle, SearchBox, GiscusComments, NewsletterForm, MobileNav.
- **`next/image` với `priority`** — ảnh cover bài viết (LCP element) có `priority` prop.
- **`next/font` self-hosted** — font load từ cùng domain, không cần DNS lookup thêm.
- **Lazy-mount Giscus** — `next/dynamic` + IntersectionObserver, load khi người đọc cuộn đến khu vực bình luận.
- **Reserve không gian ảnh** — mọi ảnh có `width` + `height` hoặc `aspect-ratio` để tránh CLS.

### 8.6 PWA Lifecycle

```
First visit:
  Browser → load page → register SW → SW installs (precache app shell)

Subsequent visit:
  Browser → SW intercept → stale-while-revalidate → serve cached + update bg

New deploy:
  Vercel deploy → new SW hash → browser detect → skipWaiting → reload → new version active

Offline (cached post):
  Browser → SW intercept → cache hit → serve from cache

Offline (uncached post):
  Browser → SW intercept → cache miss → network fail → serve /~offline fallback
```

### AD-03: Sync Notion → MDX → Git; Build chỉ đọc local MDX

- **Status:** Accepted
- **Date:** 2026-06-14
- **Deciders:** [@author]

#### Context
REQ-BUILD-001 yêu cầu build không phụ thuộc Notion API. Kinh nghiệm thực tế: trước đây fetch Notion trực tiếp lúc build dẫn đến timeout — build thất bại khi Notion API chậm/không khả dụng. Cần tách "lấy nội dung" khỏi "build website".

#### Options Considered

| Option | Pros | Cons |
|--------|------|------|
| Fetch Notion trực tiếp lúc build | Đơn giản nhất; nội dung luôn fresh | Build phụ thuộc Notion API; timeout là rủi ro thực tế; build không deterministic |
| ISR (Incremental Static Regeneration) | Vừa static vừa có thể revalidate | Vẫn gọi Notion API lúc revalidate; không giải quyết được timeout gốc |
| **Sync Notion → MDX → Git** (chosen) | Build 100% từ local file; deterministic; offline-capable | Cần sync pipeline riêng; có độ trễ giữa edit và update (cover bởi AD-04) |

#### Decision
Sync pipeline convert Notion content thành MDX files và commit vào Git repository. `next build` chỉ đọc MDX files local — không gọi Notion API.

#### Rationale
Đây là quyết định kiến trúc quan trọng nhất của project. Tách "data sync" khỏi "build" giải quyết triệt để vấn đề timeout. Build trở thành deterministic và reproducible — nếu build lỗi, lý do nằm trong code, không phải trong Notion API. Trade-off duy nhất (cần sync pipeline riêng) được xử lý bởi AD-04.

**Atomic sync:** Mỗi `syncPost` là đơn vị atomic. Lỗi ở bất kỳ bước (fetch, convert, image upload, commit) → abort toàn bài → không ghi file bán phần → MDX cũ vẫn nguyên.

#### Consequences
- **Positive:** Build deterministic, không bao giờ fail vì Notion; content có thể offline trong repo
- **Negative:** Content freshness phụ thuộc sync frequency; cần maintain sync pipeline
- **Neutral:** MDX files trong Git = audit trail tự nhiên cho nội dung

### AD-04: Dual-trigger Sync — Notion Webhook + Vercel Cron

- **Status:** Accepted
- **Date:** 2026-06-14
- **Deciders:** [@author]

#### Context
Sau khi chọn sync pipeline (AD-03), cần quyết định khi nào sync chạy. REQ-FUNC-014 yêu cầu bài mới lên web trong vài phút (real-time); REQ-FUNC-015 yêu cầu bài sửa phản ánh trong 24h (periodic). Hai yêu cầu này có trade-off khác nhau.

#### Options Considered

| Option | Pros | Cons |
|--------|------|------|
| Cron-only (mỗi 5 phút) | Đơn giản; không cần webhook | Cron 5 phút tốn quota Vercel; bài vẫn có thể trễ ≤ 5 phút; tốn function invocations |
| Webhook-only | Real-time; chỉ chạy khi cần | Notion webhook không đảm bảo delivery 100%; bài chỉnh block-level có thể không trigger webhook |
| **Webhook + Daily Cron** (chosen) | Webhook cho real-time publish; cron cho consistency | Phức tạp hơn một chút; cần implement cả hai |

#### Decision
Dùng Notion Webhook làm primary trigger (real-time); Vercel Cron chạy 1 lần/ngày lúc 2:00 AM làm lưới an toàn.

#### Rationale
Webhook đáp ứng REQ-FUNC-014 (vài phút). Daily cron đáp ứng REQ-FUNC-015 (≤ 24h) và bắt các trường hợp webhook miss (block-level edit, delivery failure). Daily cron dùng ít invocation hơn cron 5 phút, nằm trong free tier. Trade-off: phức tạp hơn một chút — chấp nhận được vì cả hai handlers dùng chung logic `syncPost`/`syncAll`.

**Optimization `last_edited_time`:** Notion Database Query API hỗ trợ filter `last_edited_time.after` — `syncAll()` chỉ fetch bài có `last_edited_time > sync:last_run` (lưu trong KV) thay vì fetch toàn bộ rồi so sánh từng bài. Giảm đáng kể số API call khi có nhiều bài.

#### Consequences
- **Positive:** Bài publish → web ≤ 5 phút; consistency đảm bảo hằng ngày; `last_edited_time` filter giảm tải Notion API call
- **Negative:** Bài chỉnh sửa sau khi publish (ngoài trường hợp webhook catch) có thể trễ đến 24h
- **Neutral:** `sync:last_run` lưu trong KV; cần xử lý trường hợp lần đầu chạy (không có `sync:last_run` → fetch all)

### AD-11: Manual sync trigger qua public API với static secret + mutex lock

- **Status:** Accepted
- **Date:** 2026-06-14
- **Deciders:** [@author]

#### Context
Cần cách trigger `syncAll()` thủ công từ bên ngoài (điện thoại, script) mà không phụ thuộc vào Vercel Cron schedule hay Vercel dashboard. Ngoài ra, Notion Webhook và Cron có thể trigger đồng thời → cần cơ chế chặn concurrent sync để tránh race condition và spam.

#### Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **Static secret + mutex (KV lock)** (chosen) | Đơn giản; không cần setup thêm; KV đã có sẵn | Secret không hết hạn nếu bị lộ (giảm thiểu bằng mutex: mỗi lần gọi chỉ chạy 1 sync) |
| TOTP (Time-based OTP) | Code hết hạn sau 30s — chặn replay attack | Cần manage TOTP seed; cần app Authenticator; phức tạp hơn nhiều |
| Webhook-only (không có manual trigger) | Đơn giản nhất | Không linh hoạt; phụ thuộc hoàn toàn vào Notion Webhook delivery |
| Admin UI screen | User-friendly; có thể hiện log | Cần build + protect UI riêng; overkill cho 1 tác giả |

#### Decision
Thêm public endpoint `/api/sync` nhận POST với `Authorization: Bearer <SYNC_SECRET>`; dùng Vercel KV làm mutex lock để chặn concurrent runs.

**Luồng mutex:**
```
POST /api/sync
  1. Verify Authorization header (Bearer SYNC_SECRET)
     → sai: 401
  2. KV.get("sync:running")
     → tồn tại: 409 Conflict {"error": "Sync đang chạy"}
  3. KV.set("sync:running", "1", { ex: 120 })   // TTL 120s — auto-expire nếu crash
  4. syncAll()
  5. KV.set("sync:last_run", ISO_timestamp)
  6. KV.del("sync:running")
  7. 200 OK
```

TTL 120s trên lock đảm bảo lock tự giải phóng nếu Function crash trước bước 6.

#### Rationale
Static secret đủ cho use case blog cá nhân — chỉ 1 người dùng endpoint này. Mutex lock giải quyết vấn đề quan trọng hơn: tránh sync chồng nhau khi webhook + cron + manual trigger cùng xảy ra. TOTP phức tạp hơn mà lợi ích không đủ lớn cho blog 1 người. Admin UI overkill.

`SYNC_SECRET` khác với `CRON_SECRET` để có thể revoke độc lập nếu cần.

#### Consequences
- **Positive:** Trigger sync từ bất cứ đâu (curl, Shortcut app trên iPhone); mutex chặn mọi concurrent run kể cả giữa webhook + cron + manual
- **Negative:** Static secret không expire — nếu lộ cần đổi manually trong Vercel env; mutex TTL 120s là ước tính, cần đủ lớn hơn max sync time
- **Neutral:** `sync:last_run` và `sync:running` là 2 KV keys mới — tổng KV keys vẫn rất nhỏ

| Term | Definition |

| App Router | Kiến trúc routing của Next.js 13+ dùng `app/` directory, hỗ trợ Server Components |

| Cron trigger | HTTP request từ Vercel Cron định kỳ để trigger `syncAll()` |

| Geist | Design system của Vercel — tối giản, đen trắng chủ đạo |
| Idempotent | Thực hiện nhiều lần với cùng input → cùng kết quả, không có side effect thừa |
| MDX | Markdown + JSX — format file lưu nội dung bài sau khi sync từ Notion |
| Pagefind | Thư viện tìm kiếm static — build index lúc compile, query hoàn toàn client-side |
| Precache | Service Worker download và cache trước app shell khi install |
| Runtime cache | Service Worker cache response khi người dùng visit (khác precache) |

| syncPost | Function đồng bộ 1 bài từ Notion: fetch → convert → images → commit |
| Vercel KV | Redis-compatible key-value store trên Vercel platform |
| Webhook | HTTP callback từ Notion khi bài thay đổi, trigger `syncPost` |
